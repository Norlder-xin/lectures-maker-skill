# Lectures Maker Skill

一个用于 Codex 的讲义制作 Skill，面向高校课程、衔接课、课堂讲义、学生练习单和教学资料整理场景。它把“生成讲义初稿 -> 根据资料和反馈迭代 -> Git 版本管理 -> 最终导出 Word”的流程固化成可复用工作流。

## 适用场景

当你希望 Codex 帮你完成以下任务时，可以使用本 Skill：

- 生成课程讲义 Markdown 初稿
- 把教师备课资料改成学生讲义
- 根据图片、PDF、Word、Markdown 或习题截图补充讲义
- 将新增题目自动插入到合适知识点，而不是简单追加到末尾
- 按要求定向修改某一节，保留其余部分不变
- 为每次讲义修改自动记录版本和修改说明
- 使用 Git 保存完整历史，支持回退、分支和最终发布 tag
- 在确认完成后导出排版清晰的 `.docx`

## 安装方法

将本仓库克隆到 Codex 的 skills 目录中。

Windows 示例：

```powershell
cd F:\codex\.codex\skills
git clone https://github.com/Norlder-xin/lectures-maker-skill.git lectures-maker
```

如果你的 Codex skills 目录在其他位置，请把仓库放到对应目录下，并确保目录结构类似：

```text
skills/
└── lectures-maker/
    ├── SKILL.md
    ├── README.md
    └── agents/
        └── openai.yaml
```

重启 Codex 或开启新会话后，即可通过 `$lectures-maker` 调用。

## 快速开始

示例提示：

```text
$lectures-maker

课程名称：高中数学衔接课
本节课主题：从 AI 潮流看初高中数学思维变化
授课对象：初升高学生
教学时长：40分钟
教学要求：以兴趣引入为主，简单涉及集合知识，生成学生使用的讲义。
```

继续修改时可以说：

```text
请把这份讲义改得更像学生使用的课堂讲义，语言简洁一些，其余没有提到的部分不要改。
```

上传资料后可以说：

```text
请分析这张习题截图，把题目插入到讲义合适位置，不要直接追加到末尾。
```

最终确认时可以说：

```text
基本没有问题了，请转换排版成 docx，保留一定空白区域给学生做笔记。
```

## 工作流程

### 1. 生成 Markdown 初稿

Skill 会先生成 Markdown，而不是直接生成 Word。默认结构包括：

```text
课程标题
学习目标
知识框架
正文内容
案例分析
课堂练习
课后习题
总结
```

学生讲义会尽量保持简洁，优先使用：

- 明确概念
- 对比表格
- 典型例题
- 填空区域
- 随堂练习
- 课后任务

### 2. 分析上传材料

当用户上传图片、PDF、Word、Markdown 或习题截图时，Skill 会提取：

```text
标题
文字
公式
图表
题目
答案
知识点
```

并整理为：

```text
新增知识点列表
新增习题列表
新增案例列表
```

新增内容会按语义插入：

| 内容类型 | 插入位置 |
|---|---|
| 概念题 | 对应知识点之后 |
| 计算题 | 对应公式之后 |
| 综合题 | 案例分析或课后任务 |
| 真题/检测题 | 课堂练习或课后习题 |
| 图片/表格 | 对应说明附近 |

### 3. 定向修改

当用户要求删除、扩写、降低难度、增加案例、增加习题或修改授课对象时，Skill 会：

1. 读取最新 Markdown 版本；
2. 只修改相关章节；
3. 保留未提到内容不变；
4. 更新标题层级、编号和目录；
5. 生成新版本；
6. 更新 `change_log.md`；
7. 提交 Git commit。

如果用户明确说“其余没有提到的部分禁止修改”，Skill 会执行局部替换，避免重写全文。

### 4. 最终导出 DOCX

只有当用户确认讲义完成时，才会导出 Word 文件。典型触发语：

```text
讲义确认完成
没有问题了
基本没有什么问题了，请转换成 docx
```

Word 输出要求：

- 一级标题使用 Heading 1
- 二级标题使用 Heading 2
- 自动目录字段
- 图片居中
- 表格美化
- 页码
- 页眉显示课程或讲义名称
- 保留学生做笔记的空白区域

## Git 版本管理规则

所有讲义项目必须使用 Git。

每次生成或修改讲义后，Skill 会执行：

```bash
git add <changed-files>
git commit -m "<auto-generated message>"
```

并维护：

```text
change_log.md
```

版本管理能力包括：

- 每次修改自动提交
- commit message 根据修改内容生成
- 保留完整历史记录
- 支持回退任意版本
- 支持用分支管理不同教学版本
- 最终发布时自动创建 tag

分支命名示例：

```text
teaching-version/<lecture-title>-advanced
teaching-version/<lecture-title>-student-handout
```

最终 tag 示例：

```text
<safe-lecture-title>-final-v1
```

## 生成文件示例

新项目优先使用讲义标题命名：

```text
第1讲-AI时代我们为什么要学高中数学_draft.md
第1讲-AI时代我们为什么要学高中数学_v1.md
第1讲-AI时代我们为什么要学高中数学_v2.md
第1讲-AI时代我们为什么要学高中数学_final.docx
change_log.md
```

如果现有项目已经使用通用命名，也会继续沿用：

```text
lecture_draft.md
lecture_v1.md
lecture_v2.md
lecture_final.docx
change_log.md
```

## change_log.md 格式

每次修改会追加一条记录：

```markdown
## YYYY-MM-DD

- 版本：`lecture_v2.md`
- 修改章节：第四节、第五节、第八节
- 修改原因：根据用户要求扩充案例并替换练习题
- 插入位置说明：新增题目插入到“随堂练习”对应知识点之后
- Git提交：`abc1234`
```

## 注意事项

- 永远先生成 Markdown，再生成 Word。
- 不要把新增内容简单追加到文档末尾。
- 不要丢失用户上传或指定保留的内容。
- 不要在用户要求局部修改时重写全文。
- 最终 DOCX 需要基于最新 Markdown 版本生成。
- 若 DOCX 渲染工具不可用，应进行结构检查并明确告知用户视觉渲染 QA 未完成。
